# MedSim Backend

Backend-часть платформы `MedSim` для учебной ролевой симуляции процессов медицинской организации.

## Что сейчас умеет backend

На текущем этапе backend покрывает уже два связанных сценария: вход игроков в игровую комнату и рабочее пространство ведущего с базовым управлением сессией.

Что уже реализовано:

- запуск `Spring Boot` приложения;
- подключение к `PostgreSQL`;
- применение миграций через `Flyway`;
- вход игрока в игровую сессию;
- `Basic Auth` для тестового ведущего;
- просмотр списка активных сессий;
- просмотр состава конкретной сессии;
- настройка этапов игровой сессии;
- ручное и автоматическое назначение игровых ролей;
- запуск, завершение и удаление сессий.

## Основные сценарии

### 1. Игрок подключается к сессии

1. Игрок вводит имя, должность и код сессии на фронте.
2. Фронт вызывает `POST /api/player-sessions/join`.
3. Backend:
   - создаёт или переиспользует игрока;
   - создаёт или переиспользует игровую сессию;
   - создаёт связь игрока с сессией;
   - возвращает данные для стартовой комнаты ожидания.

### 2. Ведущий настраивает и управляет сессией

Ведущий:

- проходит проверку логина и пароля через `Basic Auth`;
- получает свой профиль через `GET /api/auth/me`;
- видит список игровых сессий;
- открывает конкретную сессию и состав участников;
- задаёт количество этапов, время и режим каждого этапа;
- вручную назначает роли участникам;
- выполняет автоматическое распределение ролей;
- запускает, завершает и удаляет сессию.

## Логика настройки этапов и ролей

### Настройки этапов

Для каждой игровой сессии теперь сохраняются:

- номер этапа;
- длительность этапа в минутах;
- режим взаимодействия:
  - `CHAT_ONLY`
  - `CHAT_AND_KANBAN`

Настройки этапов можно изменять только пока сессия находится в статусе `LOBBY`.

### Игровые роли

Поддерживаются два варианта назначения ролей:

- ручное назначение любой роли ведущим;
- автоматическое распределение.

Ограничения для автоматического распределения:

- требуется минимум `3` участника;
- обязательно назначаются роли:
  - `Главный врач`
  - `Главная медсестра`
  - `Главный инженер`
- участник не получает игровую роль, совпадающую с его реальной должностью;
- остальные участники получают роль `Исполнитель`.

Изменение ролей также разрешено только до старта игры, пока сессия остаётся в `LOBBY`.

## Текущая модель данных

Сейчас в базе используются таблицы:

- `users`
  Для системных пользователей, таких как ведущий и суперпользователь.
- `players`
  Для обычных участников игры.
- `game_sessions`
  Для игровых сессий/комнат.
- `session_participants`
  Для связи игрока с конкретной игровой сессией и назначенной игровой ролью.
- `session_stage_settings`
  Для настроек этапов конкретной сессии.

Дополнительно:

- на уровне БД есть защита от дублей игроков по паре `display_name + hospital_position`;
- на уровне БД есть уникальность связи `game_session_id + player_id`;
- на уровне БД есть уникальность пары `game_session_id + stage_number`.

## Текущие endpoints

### Публичные

#### `POST /api/player-sessions/join`

Подключает игрока к сессии и создаёт необходимые записи, если их ещё нет.

Пример запроса:

```json
{
  "displayName": "Анна Петрова",
  "hospitalPosition": "Главная медсестра",
  "sessionCode": "WARD-12"
}
```

### Защищённые для ведущего

#### `GET /api/auth/me`

Возвращает текущего аутентифицированного системного пользователя.

#### `GET /api/game-sessions`

Возвращает список активных сессий с количеством участников и количеством этапов.

#### `GET /api/game-sessions/{sessionCode}/participants`

Возвращает обзор конкретной сессии, список участников и настройки этапов.

#### `PUT /api/game-sessions/{sessionCode}/stages`

Сохраняет набор этапов для сессии.

#### `POST /api/game-sessions/{sessionCode}/roles/random`

Выполняет автоматическое распределение ролей по участникам.

#### `PATCH /api/game-sessions/{sessionCode}/participants/{participantId}/role`

Назначает конкретную игровую роль выбранному участнику вручную.

#### `PATCH /api/game-sessions/{sessionCode}/start`

Переводит сессию в статус `IN_PROGRESS`.

#### `PATCH /api/game-sessions/{sessionCode}/finish`

Переводит сессию в статус `FINISHED`.

#### `DELETE /api/game-sessions/{sessionCode}`

Удаляет сессию и её участников. Игроки удаляются только если больше нигде не используются.

## Текущие ограничения

Сейчас используется временная схема служебного входа:

- есть тестовый ведущий, добавленный миграцией;
- superuser пока не реализован;
- отдельной полноценной auth-модели с токенами ещё нет.

Также код сессии пока работает как код комнаты:

- если сессия с таким кодом есть, игрок подключается к ней;
- если нет, она создаётся автоматически.
