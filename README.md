# MedSim Backend

Backend-часть платформы `MedSim` для учебной ролевой симуляции процессов медицинской организации.

## Что сейчас умеет backend

На текущем этапе backend покрывает четыре связанных сценария: подключение игроков к существующим игровым комнатам, отдельный командный workspace игрока после старта, рабочее пространство ведущего с созданием и управлением сессиями, а также мониторинг соревнующихся команд внутри сессии.

Что уже реализовано:

- запуск `Spring Boot` приложения;
- подключение к `PostgreSQL`;
- применение миграций через `Flyway`;
- публичный список доступных для игроков сессий;
- вход игрока только в существующую игровую сессию;
- `Basic Auth` для тестового ведущего;
- создание игровых сессий ведущим с автоматической генерацией уникального кода;
- указание количества команд при создании сессии;
- автоматическое создание команд со случайными русскими названиями в больничной тематике;
- переименование названия существующей сессии ведущим;
- переименование команд внутри сессии;
- ручное и автоматическое распределение игроков по командам;
- просмотр списка активных сессий;
- просмотр состава конкретной сессии;
- настройка этапов игровой сессии;
- ручное и автоматическое назначение игровых ролей;
- запуск, завершение и удаление сессий;
- отдельный endpoint командного workspace игрока после старта сессии.

## Основные сценарии

### 1. Игрок подключается к сессии

1. Игрок открывает список доступных сессий на фронте.
2. Фронт вызывает `GET /api/player-sessions/available`.
3. Игрок выбирает одну из сессий и отправляет форму.
4. Фронт вызывает `POST /api/player-sessions/join`.
5. Backend:
   - проверяет, что сессия существует;
   - проверяет, что она ещё находится в статусе `LOBBY`;
   - создаёт или переиспользует игрока;
   - создаёт или переиспользует связь игрока с сессией;
   - возвращает данные для стартовой комнаты ожидания.
6. После запуска игры фронт вызывает `GET /api/player-sessions/{sessionCode}/participants/{participantId}/workspace`.
7. Backend возвращает только состояние команды этого игрока: его роль, команду, состав его команды и настройки этапов.

### 2. Ведущий создаёт и настраивает сессию

Ведущий:

- проходит проверку логина и пароля через `Basic Auth`;
- получает свой профиль через `GET /api/auth/me`;
- создаёт игровую сессию по названию и количеству команд;
- получает автоматически сгенерированный код комнаты;
- получает автоматически созданные команды со случайными названиями;
- при необходимости меняет название сессии и названия команд;
- вручную распределяет игроков по командам или запускает полное автоматическое перераспределение;
- задаёт количество этапов, время и режим каждого этапа;
- вручную назначает роли участникам, уже распределённым по командам;
- выполняет автоматическое распределение ролей отдельно внутри каждой команды;
- запускает, завершает и удаляет сессию;
- после старта продолжает получать сводку по командам через обзор сессии, не раскрывая игрокам чужие команды.

## Логика команд, этапов и ролей

### Команды

Для каждой игровой сессии backend хранит набор команд с порядком и названием.

Правила работы с командами:

- количество команд задаётся при создании сессии;
- код сессии генерируется автоматически в формате `ABCD-12`;
- команды создаются автоматически со случайными русскими названиями в медицинской тематике;
- названия команд можно менять вручную, пока сессия в статусе `LOBBY`;
- игрока можно назначить в любую команду вручную;
- автоматическое распределение по командам при каждом запуске полностью пересобирает состав команд и выравнивает участников заново максимально поровну;
- роли можно назначать только тем участникам, у которых уже выбрана команда;
- автоматическое назначение ролей работает отдельно внутри каждой команды.

### Настройки этапов

Для каждой игровой сессии сохраняются:

- номер этапа;
- длительность этапа в минутах;
- режим взаимодействия:
  - `CHAT_ONLY`
  - `CHAT_AND_KANBAN`

Настройки этапов можно изменять только пока сессия находится в статусе `LOBBY`.

### Игровые роли

В системе используются роли:

- `Главный врач`
- `Главная медсестра`
- `Главный инженер`
- `Сестра поликлинического отделения`
- `Сестра диагностического отделения`
- `Заместитель главного инженера по медтехнике`
- `Заместитель главного инженера по АХЧ`

Поддерживаются два варианта назначения ролей:

- ручное назначение любой роли ведущим, но только участнику с командой;
- автоматическое распределение.

Ограничения для автоматического распределения:

- в каждой команде требуется минимум `3` участника, уже распределённых по командам;
- в каждой команде обязательно назначаются роли:
  - `Главный врач`
  - `Главная медсестра`
  - `Главный инженер`
- участник не получает игровую роль, совпадающую с его реальной должностью;
- оставшиеся участники внутри этой же команды получают одну из исполнительских ролей:
  - `Сестра поликлинического отделения`
  - `Сестра диагностического отделения`
  - `Заместитель главного инженера по медтехнике`
  - `Заместитель главного инженера по АХЧ`
- внутри одной команды игровые роли не повторяются;
- если в какой-то команде меньше `3` участников или больше `7`, backend возвращает `409`, потому что выполнить корректное уникальное распределение уже нельзя.

Изменение команд, этапов и ролей разрешено только до старта игры, пока сессия остаётся в `LOBBY`.

## Изоляция команд после старта

После запуска сессии backend поддерживает разделение интерфейсов по командам:

- игрок получает отдельный workspace только своей команды;
- состав других команд игроку не возвращается;
- ведущий продолжает видеть общий обзор сессии и может переключаться между командами на фронте, используя уже существующий endpoint обзора участников.

Это создаёт основу для следующего шага: командные чаты, отдельные канбан-доски и независимое игровое состояние каждой команды.

## Текущая модель данных

Сейчас в базе используются таблицы:

- `users` — системные пользователи, такие как ведущий и суперпользователь;
- `players` — обычные участники игры;
- `game_sessions` — игровые сессии/комнаты;
- `session_teams` — команды внутри игровой сессии;
- `session_participants` — связь игрока с конкретной игровой сессией, командой и назначенной игровой ролью;
- `session_stage_settings` — настройки этапов конкретной сессии.

Дополнительно:

- на уровне БД есть защита от дублей игроков по паре `display_name + hospital_position`;
- на уровне БД есть уникальность связи `game_session_id + player_id`;
- на уровне БД есть уникальность пары `game_session_id + stage_number`;
- код игровой сессии уникален и используется как публичный идентификатор комнаты;
- код генерируется backend автоматически в коротком формате `ABCD-12` и проверяется на уникальность до сохранения.

## Переменные окружения

Используются переменные из `.env`:

```env
DB_NAME=medsim
DB_USERNAME=medsim_user
DB_PASSWORD=medsim_password
DB_PORT=5433
SERVER_PORT=8080
DB_URL=jdbc:postgresql://localhost:5433/medsim
```

Важно: в локальном окружении проекта БД вынесена на порт `5433`, потому что на машине может уже работать другой `PostgreSQL` на `5432`.

## Текущие endpoints

### Публичные

#### `GET /api/player-sessions/available`

Возвращает список доступных для подключения сессий в статусе `LOBBY`.

#### `POST /api/player-sessions/join`

Подключает игрока к уже существующей сессии.

### Для ведущего

Все endpoints ниже требуют `Basic Auth`.

#### `GET /api/auth/me`

Возвращает профиль текущего системного пользователя.

#### `GET /api/game-sessions`

Возвращает список сессий для панели ведущего.

#### `POST /api/game-sessions`

Создаёт новую сессию по названию и количеству команд.

#### `PATCH /api/game-sessions/{sessionCode}/name`

Переименовывает сессию.

#### `GET /api/game-sessions/{sessionCode}/participants`

Возвращает состав сессии, команды и этапы.

#### `PATCH /api/game-sessions/{sessionCode}/teams/{teamId}/name`

Переименовывает команду внутри сессии.

#### `POST /api/game-sessions/{sessionCode}/teams/auto-assign`

При каждом запуске полностью перераспределяет игроков по командам и выравнивает состав заново.

#### `PATCH /api/game-sessions/{sessionCode}/participants/{participantId}/team`

Назначает участнику команду вручную.

#### `PUT /api/game-sessions/{sessionCode}/stages`

Сохраняет настройки этапов.

#### `POST /api/game-sessions/{sessionCode}/roles/random`

Выполняет автоматическое распределение ролей.

#### `PATCH /api/game-sessions/{sessionCode}/participants/{participantId}/role`

Назначает игровую роль вручную.

#### `PATCH /api/game-sessions/{sessionCode}/start`

Запускает сессию.

#### `PATCH /api/game-sessions/{sessionCode}/finish`

Завершает сессию.

#### `DELETE /api/game-sessions/{sessionCode}`

Удаляет сессию вместе с командами, участниками и связанными настройками.
