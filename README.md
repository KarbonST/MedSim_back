# MedSim Backend

Backend-часть платформы `MedSim` для учебной ролевой симуляции процессов медицинской организации.

## Что сейчас умеет backend

На текущем этапе backend покрывает четыре связанных сценария: подключение игроков к существующим игровым комнатам, отдельный командный workspace игрока после старта, рабочее пространство ведущего с созданием и управлением сессиями, а также мониторинг соревнующихся команд внутри сессии.

Что уже реализовано:

- запуск `Spring Boot` приложения;
- подключение к `PostgreSQL`;
- применение миграций через `Flyway`;
- публичный список доступных для игроков сессий;
- вход игрока только в существующую игровую сессию;
- `Basic Auth` для тестового ведущего;
- создание игровых сессий ведущим с автоматической генерацией уникального кода;
- указание количества команд при создании сессии;
- автоматическое создание команд со случайными русскими названиями в больничной тематике;
- переименование названия существующей сессии ведущим;
- переименование команд внутри сессии;
- ручное и автоматическое распределение игроков по командам;
- просмотр списка активных сессий;
- просмотр состава конкретной сессии;
- настройка этапов игровой сессии;
- ручное и автоматическое назначение игровых ролей;
- запуск, пауза, продолжение, завершение и удаление сессий;
- жёсткая backend-проверка готовности сессии перед стартом игры;
- отдельный endpoint командного workspace игрока после старта сессии;
- единое backend-состояние текущего этапа и таймера, общее для ведущего и игроков;
- история сообщений командного чата с доставкой новых сообщений через `WebSocket`;
- отдельный чат команды для игрока и обзор обоих командных чатов для ведущего;
- стартовый склад каждой команды с рандомным набором предметов из таблицы поликлиники;
- доступ к складу команды только для участников с руководящими игровыми ролями;
- инициализация экономического состояния каждой сессии при создании комнаты;
- отдельный каталог кабинетов поликлиники и типовых проблем, подготовленный под игровую экономику;
- настраиваемый стартовый бюджет и настраиваемый ресурс времени этапа для каждой команды;
- отдельный backend-endpoint обзора экономики сессии для ведущего.

## Основные сценарии

### 1. Игрок подключается к сессии

1. Игрок открывает список доступных сессий на фронте.
2. Фронт вызывает `GET /api/player-sessions/available`.
3. Игрок выбирает одну из сессий или вручную вводит код комнаты и отправляет форму.
4. Фронт вызывает `POST /api/player-sessions/join`.
5. Backend:
   - проверяет, что сессия существует;
   - если сессия находится в `LOBBY`, пускает нового или уже существующего участника;
   - если сессия уже началась, разрешает вход только тому участнику, который уже был в этой сессии раньше под теми же именем и должностью;
   - создаёт или переиспользует игрока;
   - создаёт или переиспользует связь игрока с сессией;
   - возвращает данные для стартовой комнаты ожидания или для повторного входа.
6. После запуска игры фронт вызывает `GET /api/player-sessions/{sessionCode}/participants/{participantId}/workspace`.
7. Backend возвращает только состояние команды этого игрока: его роль, команду, состав его команды, настройки этапов, текущее runtime-состояние игры и склад команды, если игроку назначена руководящая роль.
8. Для истории переписки команды фронт вызывает `GET /api/player-sessions/{sessionCode}/participants/{participantId}/chat`.
9. Для realtime-сообщений игрок подключается к `WebSocket /ws/team-chat` и получает только поток своей команды.

### 2. Ведущий создаёт и настраивает сессию

Ведущий:

- проходит проверку логина и пароля через `Basic Auth`;
- получает свой профиль через `GET /api/auth/me`;
- создаёт игровую сессию по названию, количеству команд, стартовому бюджету и ресурсу времени на этап;
- получает автоматически сгенерированный код комнаты;
- получает автоматически созданные команды со случайными названиями;
- при необходимости меняет название сессии и названия команд;
- вручную распределяет игроков по командам или запускает полное автоматическое перераспределение;
- задаёт количество этапов, время и режим каждого этапа;
- задаёт для новой сессии стартовый бюджет и ресурс времени на этап или оставляет дефолтные значения `15.00` и `15`;
- при необходимости обновляет эти экономические настройки позже, пока сессия остаётся в `LOBBY`;
- получает для каждой команды одинаковые стартовые экономические условия: стартовый бюджет, стартовый склад и ресурс времени этапа;
- вручную назначает роли участникам, уже распределённым по командам;
- выполняет автоматическое распределение ролей отдельно внутри каждой команды;
- запускает, ставит на паузу, продолжает, завершает и удаляет сессию;
- после старта переключает текущий этап и управляет общим таймером этапа;
- может стартовать игру только после полной подготовки команд, ролей и этапов;
- после старта продолжает получать сводку по командам через обзор сессии, не раскрывая игрокам чужие команды;
- может просматривать оба командных чата через `GET /api/game-sessions/{sessionCode}/team-chats`;
- получает новые сообщения команд в реальном времени через `WebSocket /ws/team-chat`.

## Логика команд, этапов и ролей

### Команды

Для каждой игровой сессии backend хранит набор команд с порядком и названием.

Правила работы с командами:

- количество команд задаётся при создании сессии;
- код сессии генерируется автоматически в формате `ABCD-12`;
- команды создаются автоматически со случайными русскими названиями в медицинской тематике;
- названия команд можно менять вручную, пока сессия в статусе `LOBBY`;
- игрока можно назначить в любую команду вручную;
- автоматическое распределение по командам при каждом запуске полностью пересобирает состав команд и выравнивает участников заново максимально поровну;
- при любом перераспределении по командам ранее назначенные игровые роли сбрасываются, чтобы не сохранять неконсистентный состав;
- роли можно назначать только тем участникам, у которых уже выбрана команда;
- автоматическое назначение ролей работает отдельно внутри каждой команды.

Если ведущий меняет команды после назначения ролей:

- при ручном переводе участника в другую команду все игровые роли в сессии сбрасываются;
- при полном автораспределении по командам игровые роли тоже сбрасываются;
- после этого роли нужно назначить заново, чтобы в каждой команде снова гарантировались руководящие должности.

### Настройки этапов

Для каждой игровой сессии сохраняются:

- номер этапа;
- длительность этапа в минутах;
- режим взаимодействия:
  - `CHAT_ONLY`
  - `CHAT_AND_KANBAN`

Настройки этапов можно изменять только пока сессия находится в статусе `LOBBY`.

### Экономика сессии

Backend теперь подготавливает отдельный экономический слой для каждой игровой сессии.

Что создаётся автоматически при создании комнаты:

- настройки экономики сессии;
- экономическое состояние каждой команды;
- состояние всех кабинетов поликлиники для каждой команды;
- состояние всех типовых проблем в этих кабинетах для каждой команды.

Текущие базовые предпосылки экономики:

- стартовый бюджет команды задаётся при создании сессии и по умолчанию равен `15.00`;
- ресурс времени на этап для каждой команды задаётся при создании сессии и по умолчанию равен `15` условным единицам;
- время этапа как ресурс команды хранится отдельно от реального таймера этапа;
- при выборе нового этапа ресурс времени команды сбрасывается к базовому значению;
- пока сессия не стартовала, стартовый бюджет и ресурс времени можно изменить повторно, и эти значения заново применятся ко всем командам;
- деньги и накопительные показатели команды хранятся отдельно от runtime-таймера и готовы для следующих шагов реализации.

Для каталога поликлиники backend сейчас хранит:

- кабинеты с кодом, названием, порядком и базовым доходом;
- типовые проблемы кабинетов с номером, названием, тяжестью и штрафом за игнорирование;
- тяжесть проблемы: `MINOR`, `SERIOUS`, `CRITICAL`;
- коэффициенты состояния кабинета, подготовленные под будущий расчёт дохода:
  - `MINOR` -> `0.75`
  - `SERIOUS` -> `0.40`
  - `CRITICAL` -> `0.00`

Через backend-обзор экономики ведущий уже может получить:

- стартовые экономические настройки сессии;
- текущий баланс и ресурс времени каждой команды;
- список кабинетов каждой команды;
- список проблем каждого кабинета;
- количество активных проблем в кабинете;
- худшую текущую тяжесть проблемы в кабинете;
- коэффициент состояния кабинета на основе активных проблем.

### Runtime этапа и таймера

После запуска игры backend хранит общее runtime-состояние сессии:

- текущий активный этап;
- длительность активного этапа;
- режим активного этапа;
- статус таймера: `STOPPED`, `RUNNING`, `PAUSED`;
- оставшееся время этапа в секундах;
- момент завершения таймера, если он запущен.

Это состояние едино для ведущего и всех игроков. Ведущий управляет им через отдельные runtime-endpoints, а игроки получают те же данные в `workspace` своей команды.

### Игровые роли

В системе используются роли:

- `Главный врач`
- `Главная медсестра`
- `Главный инженер`
- `Сестра поликлинического отделения`
- `Сестра диагностического отделения`
- `Заместитель главного инженера по медтехнике`
- `Заместитель главного инженера по АХЧ`

Поддерживаются два варианта назначения ролей:

- ручное назначение любой роли ведущим, но только участнику с командой;
- автоматическое распределение.

Ограничения для автоматического распределения:

- в каждой команде требуется минимум `3` участника, уже распределённых по командам;
- в каждой команде обязательно назначаются роли:
  - `Главный врач`
  - `Главная медсестра`
  - `Главный инженер`
- участник не получает игровую роль, совпадающую с его реальной должностью;
- оставшиеся участники внутри этой же команды получают одну из исполнительских ролей:
  - `Сестра поликлинического отделения`
  - `Сестра диагностического отделения`
  - `Заместитель главного инженера по медтехнике`
  - `Заместитель главного инженера по АХЧ`
- внутри одной команды игровые роли не повторяются;
- если в какой-то команде меньше `3` участников или больше `7`, backend возвращает `409`, потому что выполнить корректное уникальное распределение уже нельзя.

Изменение команд, этапов и ролей разрешено только до старта игры, пока сессия остаётся в `LOBBY`.

### Условия старта игры

Перед переводом сессии из `LOBBY` в `IN_PROGRESS` backend теперь проверяет, что:

- сохранён хотя бы один этап сессии;
- у каждого подключившегося игрока назначена команда;
- в каждой команде назначены все три обязательные руководящие роли:
  - `Главный врач`
  - `Главная медсестра`
  - `Главный инженер`

Если хотя бы одно из этих условий не выполнено, backend возвращает `409 Conflict` и не запускает игру.

### Пауза и завершение игры

После старта backend поддерживает ещё два явных действия для ведущего:

- `pause` переводит сессию в статус `PAUSED`;
- повторный `start` возобновляет игру и возвращает статус `IN_PROGRESS`;
- `finish` завершает как активную, так и поставленную на паузу игру.

Во фронте действие завершения дополнительно подтверждается через диалог, чтобы игру нельзя было закрыть случайным нажатием.

Отдельно от статуса игры backend теперь поддерживает runtime-управление этапом:

- выбор текущего этапа;
- запуск общего таймера этапа;
- паузу таймера этапа;
- сброс таймера этапа к полной длительности выбранного этапа.

## Изоляция команд после старта

После запуска сессии backend поддерживает разделение интерфейсов по командам:

- игрок получает отдельный workspace только своей команды;
- состав других команд игроку не возвращается;
- склад команды возвращается только игрокам с руководящими ролями;
- ведущий продолжает видеть общий обзор сессии и может переключаться между командами на фронте, используя уже существующий endpoint обзора участников;
- история и realtime-поток командного чата для игрока ограничены только его командой;
- ведущий получает обзор обоих командных чатов внутри одной сессии.

Это уже создаёт основу для следующего шага: отдельные канбан-доски и независимое игровое состояние каждой команды.

## Текущая модель данных

Сейчас в базе используются таблицы:

- `users` — системные пользователи, такие как ведущий и суперпользователь;
- `players` — обычные участники игры;
- `game_sessions` — игровые сессии/комнаты;
- `session_teams` — команды внутри игровой сессии;
- `session_participants` — связь игрока с конкретной игровой сессией, командой и назначенной игровой ролью;
- `session_stage_settings` — настройки этапов конкретной сессии;
- `team_chat_messages` — сообщения командных чатов внутри сессии;
- `clinic_room_templates` — справочник кабинетов поликлиники для игровой экономики;
- `clinic_room_problem_templates` — справочник типовых проблем кабинетов;
- `session_economy_settings` — стартовые экономические настройки конкретной игровой сессии;
- `team_economy_states` — текущее экономическое состояние каждой команды;
- `team_room_states` — состояние кабинетов поликлиники отдельно для каждой команды;
- `team_problem_states` — состояние проблем кабинетов отдельно для каждой команды.

Часть runtime-состояния этапа и таймера хранится прямо в `game_sessions`: активный этап, статус таймера, остаток времени и время последнего обновления.

Дополнительно:

- на уровне БД есть защита от дублей игроков по паре `display_name + hospital_position`;
- на уровне БД есть уникальность связи `game_session_id + player_id`;
- на уровне БД есть уникальность пары `game_session_id + stage_number`;
- код игровой сессии уникален и используется как публичный идентификатор комнаты;
- код генерируется backend автоматически в коротком формате `ABCD-12` и проверяется на уникальность до сохранения.

## Переменные окружения

Используются переменные из `.env`:

```env
DB_NAME=medsim
DB_USERNAME=medsim_user
DB_PASSWORD=medsim_password
DB_PORT=5433
SERVER_PORT=8080
FRONTEND_PORT=5173
DB_URL=jdbc:postgresql://localhost:5433/medsim
```

Важно: в локальном окружении проекта БД вынесена на порт `5433`, потому что на машине может уже работать другой `PostgreSQL` на `5432`.

## Как запустить

### Вариант 1. Поднять весь стек одной командой через Docker

```bash
cd /Users/mihailbykadorov/Desktop/MedSim/MedSim_back
cp .env.example .env
docker compose --env-file .env up --build -d
```

После запуска доступны:

- `http://localhost:5173` — frontend
- `http://localhost:8080` — backend API

Важно: этот compose-файл собирает не только backend и PostgreSQL, но и frontend из соседнего репозитория `../MedSim_front`. Для такого запуска `MedSim_back` и `MedSim_front` должны лежать рядом в одной родительской папке.

Остановка всего стека:

```bash
cd /Users/mihailbykadorov/Desktop/MedSim/MedSim_back
docker compose down
```

## Текущие endpoints

### Публичные

#### `GET /api/player-sessions/available`

Возвращает список доступных для подключения сессий в статусе `LOBBY`.

#### `POST /api/player-sessions/join`

Подключает игрока к уже существующей сессии. Новые участники могут входить только в `LOBBY`, а уже существующий участник может вернуться в начатую игру по тому же коду, имени и должности.

### Для ведущего

Все endpoints ниже требуют `Basic Auth`.

#### `GET /api/auth/me`

Возвращает профиль текущего системного пользователя.

#### `GET /api/game-sessions`

Возвращает список сессий для панели ведущего.

#### `POST /api/game-sessions`

Создаёт новую сессию по названию и количеству команд.

#### `PATCH /api/game-sessions/{sessionCode}/name`

Переименовывает сессию.

#### `GET /api/game-sessions/{sessionCode}/participants`

Возвращает состав сессии, команды, этапы и текущее runtime-состояние.

#### `GET /api/game-sessions/{sessionCode}/economy`

Возвращает экономические настройки сессии, баланс и ресурс времени команд, кабинеты поликлиники и текущие состояния их проблем.

#### `PUT /api/game-sessions/{sessionCode}/economy/settings`

Обновляет стартовый бюджет и ресурс времени на этап для всех команд, пока сессия остаётся в `LOBBY`. При сохранении новые значения заново применяются к экономическому состоянию команд.

#### `PATCH /api/game-sessions/{sessionCode}/teams/{teamId}/name`

Переименовывает команду внутри сессии.

#### `POST /api/game-sessions/{sessionCode}/teams/auto-assign`

При каждом запуске полностью перераспределяет игроков по командам и выравнивает состав заново.

#### `PATCH /api/game-sessions/{sessionCode}/participants/{participantId}/team`

Назначает участнику команду вручную.

#### `PUT /api/game-sessions/{sessionCode}/stages`

Сохраняет настройки этапов и инициализирует runtime первого этапа.

#### `PATCH /api/game-sessions/{sessionCode}/runtime/stage`

Переключает текущий активный этап.

#### `PATCH /api/game-sessions/{sessionCode}/runtime/timer/start`

Запускает общий таймер текущего этапа.

#### `PATCH /api/game-sessions/{sessionCode}/runtime/timer/pause`

Ставит общий таймер текущего этапа на паузу.

#### `PATCH /api/game-sessions/{sessionCode}/runtime/timer/reset`

Сбрасывает общий таймер текущего этапа к полной длительности этапа.

#### `POST /api/game-sessions/{sessionCode}/roles/random`

Выполняет автоматическое распределение ролей.

#### `PATCH /api/game-sessions/{sessionCode}/participants/{participantId}/role`

Назначает игровую роль вручную.

#### `PATCH /api/game-sessions/{sessionCode}/start`

Запускает сессию.

#### `PATCH /api/game-sessions/{sessionCode}/pause`

Переводит сессию в статус `PAUSED`.

#### `PATCH /api/game-sessions/{sessionCode}/finish`

Завершает сессию.

#### `DELETE /api/game-sessions/{sessionCode}`

Удаляет сессию вместе с командами, участниками и связанными настройками.
