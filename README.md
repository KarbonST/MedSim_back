# MedSim Backend

Backend-часть платформы `MedSim` для учебной ролевой симуляции работы медицинской организации. Сервис отвечает за игровые сессии, команды, роли, чат, канбан, экономику, склад, уведомления и историю действий по задачам.

## Быстрый Запуск

### Весь стек через Docker Compose

`docker compose` запускает `PostgreSQL`, backend и frontend из соседнего репозитория `../MedSim_front`.

```bash
cd /Users/mihailbykadorov/Desktop/MedSim/MedSim_back
cp .env.example .env
docker compose --env-file .env up --build -d
```

После запуска:

- frontend: `http://localhost:5173`
- backend API: `http://localhost:8080`

Остановка:

```bash
cd /Users/mihailbykadorov/Desktop/MedSim/MedSim_back
docker compose down
```

### Backend локально

```bash
cd /Users/mihailbykadorov/Desktop/MedSim/MedSim_back
cp .env.example .env
docker compose --env-file .env up -d postgres
set -a
source .env
set +a
mvn spring-boot:run
```

Локальный PostgreSQL в `.env.example` вынесен на порт `5433`, чтобы не конфликтовать с системным PostgreSQL на `5432`.

## Что Реализовано

- Создание игровых сессий с автоматическим кодом комнаты.
- Настройка количества команд, названий команд, стартового бюджета и ресурса времени.
- Ручное и случайное распределение игроков по командам.
- Ручное и случайное распределение игровых ролей внутри команд.
- Проверка готовности перед стартом: этапы, команды и обязательные руководящие роли.
- Единое runtime-состояние сессии: активный этап, режим этапа, таймер, пауза и завершение.
- Командный чат с историей и `WebSocket`-доставкой новых сообщений.
- Два режима этапов: задачи через чат и задачи через полноценную канбан-доску.
- Канбан-процесс с первичным разбором, распределением, исполнением и двумя уровнями согласования.
- Уведомления игрокам по канбан-событиям без отдельного real-time слоя.
- История карточки: кто выставил приоритет, назначил подразделение, взял в работу, отправил на согласование, закрыл или вернул.
- Экономика команды: деньги, время этапа, доходы, штрафы, бонусы, расходы и итоги этапов.
- Мягкое резервирование ресурсов при выборе способа решения задачи.
- Варианты решений задач, импортированные из таблицы поликлиники, плюс стандартные варианты для оставшихся задач.
- Вероятности успеха решений с учетом выбранного подразделения.
- Стартовый склад команды, одинаковый для всех команд в одной сессии.
- Автоматическая генерация стартового склада и ручная настройка склада ведущим до старта.
- Автоматическое равномерное распределение задач по этапам и ручная настройка пропорции ведущим.
- Read-only обзор канбана и экономики для ведущего.

## Игровая Логика

### Команды И Роли

При создании сессии backend создает команды со случайными медицинскими названиями. До старта ведущий может переименовывать команды, вручную переносить игроков, запускать случайное распределение по командам и назначать роли.

Обязательные роли в каждой команде:

- `Главный врач`
- `Главная медсестра`
- `Главный инженер`

Исполнительские роли:

- `Сестра поликлинического отделения`
- `Сестра диагностического отделения`
- `Заместитель главного инженера по медтехнике`
- `Заместитель главного инженера по АХЧ`

Если игрок меняет команду, назначенные игровые роли сбрасываются, чтобы состав не стал неконсистентным.

### Этапы И Задачи

Поддерживаются режимы:

- `CHAT_WITH_PROBLEMS` — задачи доступны в игровом процессе, но без полноценной визуальной канбан-доски.
- `CHAT_AND_KANBAN` — задачи доступны через канбан-доску.

Задачи распределяются по этапам на уровне конкретной сессии:

- по умолчанию backend делит все задачи максимально равномерно между этапами;
- ведущий может вручную указать, сколько задач попадет на каждый этап;
- сумма ручного распределения должна совпадать с общим количеством шаблонных проблем;
- назначенный этап хранится в `team_problem_states.stage_number`, поэтому можно менять распределение в сессии, не меняя исходный каталог проблем.

Если в сессии один этап, в него попадут все задачи. Если этапов несколько, задачи распределяются между ними автоматически или по ручной пропорции ведущего.

### Канбан

Канбан-статусы в backend:

- `REGISTERED` — задачи текущего этапа, еще не разобраны главврачом.
- `ASSIGNED` — задача разобрана главврачом и отправлена на распределение в подразделение.
- `READY_FOR_WORK` — руководитель подразделения назначил исполнителя.
- `IN_PROGRESS` — исполнитель взял задачу в работу.
- `DEPARTMENT_REVIEW` — исполнитель отправил задачу на согласование руководителю подразделения.
- `CHIEF_DOCTOR_REVIEW` — руководитель подразделения согласовал и отправил задачу главврачу.
- `REWORK` — задача возвращена после неуспешного согласования.
- `HOLD` — задача отложена до следующего подходящего этапа.
- `DONE` — задача закрыта после финального согласования главврачом.

Ключевые правила доступа:

- Главврач выставляет приоритет и ответственное подразделение.
- Главная медсестра назначает исполнителей только для сестринского подразделения.
- Главный инженер назначает исполнителей только для инженерного подразделения.
- Руководители подразделений не могут назначать задачу сами себе.
- Исполнитель выбирает способ решения, создает резерв ресурсов, берет задачу в работу и отправляет ее на согласование.
- Сначала задачу согласует руководитель подразделения, затем финально согласует главврач.
- При возврате задачи резерв ресурсов снимается, а карточка возвращается в пул задач этапа.

### Экономика И Резервы

Экономика команды хранит:

- текущий баланс;
- ресурс времени текущего этапа;
- активные резервы денег, времени и складских позиций;
- доступные ресурсы с учетом резервов;
- суммарные доходы, расходы, штрафы и бонусы;
- итоги этапов;
- последние события экономики.

Ресурсы работают по схеме мягкого резерва:

- при выборе способа решения создается резерв денег, времени и нужного предмета склада;
- если базовых ресурсов команды недостаточно, способ решения недоступен;
- параллельные резервы могут конкурировать за один и тот же ресурс;
- перед финальным закрытием главврачом backend повторно проверяет ресурсы;
- если после других согласований ресурсов уже не хватает, финальное закрытие блокируется;
- фактическое списание происходит только при статусе `DONE`;
- при возврате или отправке в `HOLD` активный резерв снимается.

Итоги этапа считаются при переключении с завершенного этапа на следующий: backend начисляет доходы кабинетов, штрафы за нерешенные проблемы и бонус за закрытые задачи этапа.

### Склад

Стартовый склад создается при создании сессии:

- набор складских позиций формируется случайно;
- один и тот же стартовый склад копируется во все команды, чтобы условия были равными;
- ведущий может вручную изменить количество каждой позиции до старта;
- ведущий может заново сгенерировать случайный склад до старта;
- количество позиции может быть `0`, чтобы предмет существовал в списке, но отсутствовал на складе.

Игроки-исполнители склад не видят. Руководящие роли получают склад в своем workspace, а ведущий видит общий стартовый список при настройке сессии.

## Данные И Миграции

Основные таблицы:

- `users` — системные пользователи.
- `players` — игроки.
- `game_sessions` — игровые сессии и runtime-состояние.
- `session_teams` — команды.
- `session_participants` — участники сессии, команда и игровая роль.
- `session_stage_settings` — этапы, длительность, режим и количество задач.
- `team_chat_messages` — командный чат.
- `team_inventory_items` — склад команды.
- `clinic_room_templates` — кабинеты поликлиники.
- `clinic_room_problem_templates` — шаблоны проблем.
- `team_room_states` — состояние кабинетов команды.
- `team_problem_states` — состояние проблем команды и назначенный этап.
- `team_kanban_cards` — карточки канбана.
- `team_kanban_card_events` — история карточек и источник уведомлений.
- `kanban_solution_options` — варианты решения задач.
- `team_resource_reservations` — резервы ресурсов.
- `session_economy_settings` — настройки экономики сессии.
- `team_economy_states` — текущее экономическое состояние команды.
- `team_economy_events` — события экономики и итоги этапов.

Миграции лежат в двух местах:

- SQL-миграции: `src/main/resources/db/migration`
- Java-миграции Flyway: `src/main/java/db/migration`

Java-миграции используются там, где нужно аккуратно обработать отличия ограничений БД или извлечь вероятности из текстовых описаний вариантов решения.
